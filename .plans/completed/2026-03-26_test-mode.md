# Test Mode — Force-block posts for UI verification

- **Date**: 2026-03-25
- **Status**: draft
- **Type**: feature

## Problem
The extension loads on LinkedIn but we can't verify the hide/roast UI works because the phrase matcher rarely triggers on real posts. We need a way to force-block specific posts to visually test the banners and hiding.

## Approach
Add a `testMode` flag to the config. When enabled, `content.js` force-blocks the 3rd and 5th post found on each scan — bypassing the detector entirely and feeding them a synthetic result with score 99 and match "Test mode". This lives entirely in `content.js`; the detector stays untouched.

Add a "Test Mode" toggle to the popup, styled distinctly (yellow/warning) so it's obviously a debug tool.

## Changes

| File | Change |
|------|--------|
| `src/content.js` | In `scanFeed`, track post index. For posts at index 2 and 4 (0-based), skip `analyzePost` and use a hardcoded `{ blocked: true, score: 99, matches: ["Test mode"] }` result when `config.testMode` is true. |
| `src/popup/popup.html` | Add a "Test Mode" toggle row below the Enabled toggle, styled with a yellow/warning color to distinguish it from real settings. |
| `src/popup/popup.js` | Wire up the test mode toggle — read/write `testMode` from `chrome.storage.sync`. Include it in `save()` and `loadState()`. |

## Tests
Manual: enable test mode in popup, reload LinkedIn feed, confirm 3rd and 5th posts show roast banners (or are hidden, depending on mode). Toggle off, reload, confirm all posts are back to normal.

## Out of Scope
- No console logging / debug panel
- No per-post "force block this one" feature
- Test mode doesn't persist blocked count to stats
